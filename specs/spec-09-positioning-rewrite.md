# Spec 09 — Positioning Rewrite

**Project:** code-auditor-mcp
**Ships as:** v4.0.1 (docs + metadata only)
**Done means:** pushed to main, published to npm, GitHub About updated.
**Depends on:** Spec 08 merged and published.

## Context

The current README leads with "AI assistants are flying blind" — a 2025 premise that's false in 2026 and positions the product against a fight the host tools already won. The GitHub About blurb is emoji marketing that undercuts credibility with the exact audience (senior engineers configuring MCP servers and agent hooks) the product serves. The product identity after Specs 01–08 is: **the policy gate for AI coding agents** — project-specific invariants and code-quality analysis enforced inside the agent's edit loop, with code intelligence as supporting infrastructure.

## Requirements

### R1 — GitHub About + npm metadata

1. GitHub About text, verbatim: `Architectural invariants and code quality analysis, enforced inside your AI agent's edit loop. MCP server + CLI + Claude Code plugin. TypeScript, JavaScript, Go.` No emoji.
2. `package.json` `description` set to the same line; `keywords` updated to: `mcp`, `code-quality`, `static-analysis`, `ai-agents`, `claude-code`, `invariants`, `sarif`, `tree-sitter`, `linter`, `solid`, plus existing language keywords.

### R2 — README rewrite

Structure, in order:

1. **Title + one-paragraph pitch.** The agent-loop framing: agents write code fast; this makes them write code that follows *your* rules. Invariants + analyzers + diff-scoped enforcement.
2. **The loop, shown not told.** A short worked sequence: `.codeauditor.json` with one real invariant rule → agent edit that violates it → hook output blocking with the rule's message. This replaces the current "Problem It Solves" section.
3. **Quick start** — three paths, in this order: Claude Code plugin (two commands, from Spec 07), generic MCP (`claude mcp add` / Cursor JSON), CLI (`npx code-audit`).
4. **Invariant rules** — the five kinds with one realistic example each (Spec 05/08 content, consolidated here as the flagship section).
5. **Analyzers** — SOLID/DRY/React/Documentation/Data Access/Schema, one line each.
6. **Diff-scoped auditing & hooks** — scopes, `code-audit changed` contract, hook recipes.
7. **Code intelligence** — search operators table, code maps, `tasks` including `from_audit`. Framed as supporting infrastructure, not the headline.
8. **CI/SARIF** — the Spec 06 workflow.
9. **Configuration, data dir, contributing, license** — carried forward, updated for the v4 tool surface.

Tone constraints: no emoji section headers, no exclamation-point marketing, no "x-ray vision," no fabricated example transcripts — every example in the README must be reproducible against the shipped version, and the R2.2 sequence is generated by actually running it.

### R3 — Consistency sweep

1. CHANGELOG entries for the whole 2.7.0 → 4.0.1 arc are present and accurate.
2. `server.json`, CONTRIBUTING.md, and all config examples reference only the v4 tool surface and entry point. `grep -rn "start_audit\|search_code\|project_tasks\|mcp-index\|mcp-standalone" README.md CONTRIBUTING.md server.json examples/ configs/` — zero matches.

## Acceptance evidence

1. Rendered README reviewed section-by-section against the R2 structure list.
2. Reproducibility: transcript showing the R2.2 worked sequence executed against the published v4.0.1 package with byte-identical output to what the README shows.
3. R3.2 grep output: zero matches.
4. GitHub About screenshot/text confirmation; `npm view code-auditor-mcp description` returns the R1 line.

## Explicitly out of scope

- Any code changes beyond examples/configs. If the rewrite exposes a behavior bug, it's reported and fixed under its own heading, not silently folded in.
- Website/landing page — the README is the landing page.
